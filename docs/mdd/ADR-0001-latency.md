# ADR-0001: выбор архитектуры по времени отклика системы (MDD)

- Статус: принято
- Дата: 2026-05-30
- Контекст оценки: Metrics Driven Development (критерий 5)

## Контекст и проблема

Даны два набора данных по времени отклика системы (latency, секунды): текущая
архитектура и предлагаемая улучшенная (по 500000 наблюдений каждая, сэмплинг из
задания: `loc=3.5/2.0`, `scale=0.4`, `seed=42`). Нужно обоснованно решить по MDD,
переходить ли на улучшенную архитектуру, а не "на глазок". Воспроизведение графика
и расчёты - в `docs/mdd/analysis.py`, визуализация - `docs/mdd/latency_comparison.png`.

## Драйверы решения

- Бизнес-SLO по времени отклика: p95 не выше 3.0 секунды.
- Практическая, а не только статистическая значимость улучшения.
- Воспроизводимость и прозрачность статистического вывода.

## Рассмотренные варианты

1. Оставить текущую архитектуру (loc=3.5с).
2. Перейти на улучшенную архитектуру (loc=2.0с).

## Решение (Decision)

### Гипотезы и тест
- H0: средние времена отклика двух архитектур равны.
- H1: средние времена отклика различаются (двусторонняя альтернатива).
- Уровень значимости: alpha = 0.05.
- Тесты: Welch t-test (не предполагает равенства дисперсий) и непараметрический
  Mann-Whitney U (контроль на случай отклонений от нормальности).

### Результаты
- Средние: текущая = 3.4995с, улучшенная = 1.9992с; разность = 1.5003с.
- Welch t-test: t = 1875.08, p < 1e-300 (практически 0). H0 отклоняется.
- Mann-Whitney U: p < 1e-300. Согласуется с t-тестом.
- Размер эффекта Cohen's d = 3.750 (очень большой; по Cohen > 0.8 уже большой).
- 95% доверительный интервал разности средних: [1.4988, 1.5019] с.
- p95: текущая = 4.157с, улучшенная = 2.657с (SLO = 3.0с).
- Доля наблюдений выше SLO: текущая = 89.46%, улучшенная = 0.625%.

### Статистическая против практической значимости (зрелость анализа)
При n = 500000 любой тест почти неизбежно даёт p, близкое к нулю: огромная выборка
делает статистически значимым даже ничтожное различие. Поэтому сам факт p < 0.05
здесь малоинформативен и НЕ может быть основанием для решения.

Решение определяется ПРАКТИЧЕСКОЙ значимостью:
- размер эффекта Cohen's d = 3.75 - разделение распределений колоссальное;
- доверительный интервал разности средних узкий и целиком вокруг 1.5с - сокращение
  времени отклика оценивается как ~1.5 секунды с высокой точностью;
- ключевой аргумент: улучшенная архитектура переводит p95 ниже SLO 3.0с
  (4.157с -> 2.657с), а долю запросов вне SLO снижает с 89.46% до 0.625%.

Именно пересечение SLO (а не p-value) делает улучшение практически значимым.

### Итоговое архитектурное решение
Переходим на улучшенную архитектуру. Обоснование: сокращение времени отклика на
~1.5с (Cohen's d = 3.75) переводит систему из устойчивого нарушения SLO (p95 4.16с,
89% вне SLO) в уверенное соблюдение (p95 2.66с, 0.6% вне SLO).

## Последствия

- Положительные: соблюдение SLO по времени отклика, лучший пользовательский опыт,
  запас по нагрузке.
- Отрицательные/риски: стоимость миграции; на проде значения latency будут отличаться
  от модельной выборки - после внедрения подтверждаем эффект на реальной телеметрии
  (Prometheus/Grafana) и пересматриваем SLO при необходимости.
- Методический вывод на будущее: при больших выборках решения принимаем по размеру
  эффекта, доверительным интервалам и попаданию в SLO, а не по факту p < 0.05.
