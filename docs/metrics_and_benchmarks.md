# Бенчмарки и метрики оценки качества модели

## Философия оценки

Оценка качества ML-модуля Uneemi строится в три слоя, соответствующих разным фазам жизненного цикла:

1. **Sanity-проверка на публичных бенчмарках** - верификация того, что предобученная модель загружена и запущена корректно, результаты воспроизводят заявленные в статье SigLIP 2.
2. **Proxy-бенчмарки на собственных данных до запуска** - оценка качества эмбеддингов применительно к задаче вайб-матчинга в отсутствие размеченных пар «пользователь A ↔ пользователь B». Используются задачи-прокси (self-consistency, cluster coherence, triplet tests), не требующие разметки исходов матчей.
3. **Online-метрики после запуска MVP** - прямое измерение бизнес-ценности модели через поведение реальных пользователей.

Ключевое ограничение - отсутствие размеченных исходов матчей. До запуска ни одна метрика не может напрямую измерить «качество матчей», поэтому proxy-слой критичен для принятия решения о готовности модели к продакшену.

## 1. Публичные бенчмарки SigLIP 2 (sanity-проверка)

Цель - убедиться, что ONNX-экспорт и pipeline инференса не вносят деградации качества относительно референсных чисел из статьи. Запускается однократно перед деплоем.

| Бенчмарк | Задача | Paper (B/16@256) | Целевой порог (наш ONNX@224) |
|---|---|---|---|
| **ImageNet-1k zero-shot** | Классификация 1000 классов | 79.1% top-1 | ≥ 75% top-1 |
| **COCO Image→Text R@1** | Image-to-text retrieval | 53.2% | ≥ 48% |
| **COCO Text→Image R@1** | Text-to-image retrieval | ~43% | ≥ 38% |
| **Flickr30k I→T R@1** | Image-to-text retrieval | ~87% | ≥ 82% |
| **XM3600 Crossmodal (RU)** | Мультиязычный retrieval (русский) | 40.7% avg R@1 | ≥ 35% |

**Источники:** Tschannen et al., arXiv:2502.14786 (Table 1, Table 7). XM3600 - основной бенчмарк, подтверждающий готовность модели к русскоязычному контенту.

**Методика запуска:** через библиотеку `clip_benchmark` (https://github.com/LAION-AI/CLIP_benchmark), совместимую с HuggingFace моделями. Прогон занимает ~2 часа на одном GPU или ~6 часов на целевом CPU-сервере.

**Критерий прохождения:** каждая метрика на нашем ONNX-экспорте должна отличаться от paper-значения не более чем на -5 п.п. Большее расхождение сигнализирует о проблеме в pipeline (неверный препроцессинг, потеря точности при экспорте, ошибка в текстовых шаблонах).

## 2. Proxy-бенчмарки на собственных данных (pre-launch)

Пять проверок, измеряющих качество эмбеддингов применительно к задаче вайб-матчинга без разметки исходов.

### 2.1. Board self-consistency (split-half test)

**Гипотеза:** если представление доски устойчиво, то центроиды двух случайных половин одной и той же доски должны быть близки.

**Методика:**
1. Собрать 100 тестовых досок по 40 изображений (члены команды + добровольцы из целевой аудитории + synthetic через Pinterest/subreddit scraping).
2. Для каждой доски: случайно разбить на две половины по 20 изображений (10 мемов + 10 эстетики в каждой).
3. Вычислить центроиды `centroid_A` и `centroid_B` с весами 0.4/0.6 (мем/эстетика).
4. Измерить `cos(centroid_A, centroid_B)` для каждой доски.

**Метрика:** `split_half_similarity = mean(cos(A, B))`

**Target:** ≥ 0.85 (медиана), ≥ 0.75 (5-й перцентиль). Низкие значения означают, что представление доски нестабильно - это потребует увеличения количества изображений или пересмотра весов фьюжна.

### 2.2. Aesthetic cluster coherence

**Гипотеза:** изображения из одного стилистического домена должны образовывать выраженные кластеры в пространстве эмбеддингов.

**Методика:**
1. Собрать датасет из 5-7 известных эстетических направлений, по 200 изображений в каждом (cottagecore, dark academia, minimalism, cyberpunk, y2k, kidcore, weirdcore - источники: соответствующие subreddits, Pinterest boards).
2. Получить эмбеддинги всех изображений через SigLIP 2.
3. Вычислить метрики разделимости.

**Метрики:**
- **Silhouette score** - оценка плотности кластеров. Target: ≥ 0.20 (для стилистической задачи, где границы размыты, значение > 0.15 считается сильным сигналом).
- **k-NN purity @ k=10** - доля ближайших 10 соседей, относящихся к тому же стилю. Target: ≥ 0.65.
- **Adjusted Rand Index** после k-means кластеризации с k = число стилей. Target: ≥ 0.35.

### 2.3. Style triplet test

**Гипотеза:** модель должна различать стилистическое сходство и поверхностное визуальное сходство (одинаковый объект в разных стилях).

**Методика:**
1. Курированно собрать 200 триплетов `(anchor, positive, negative)`, где:
   - `anchor` и `positive` - разные изображения в одном стиле (например, два «cottagecore» с разными объектами);
   - `anchor` и `negative` - одинаковый объект в разных стилях (например, «cat in cottagecore» vs «cat in cyberpunk»).
2. Для каждого триплета проверить: `cos(anchor, positive) > cos(anchor, negative)`.

**Метрика:** `triplet_accuracy = доля выполненных условий`

**Target:** ≥ 0.70. Пороговое значение на уровне случайного угадывания - 0.5; значение > 0.70 подтверждает, что модель приоритезирует стиль над объектным составом - критичное свойство для задачи вайб-матчинга.

### 2.4. Human-agreement evaluation (small-scale)

**Гипотеза:** cosine similarity между профильными векторами должна коррелировать с человеческой оценкой «похожести вайба».

**Методика:**
1. Собрать 50 пар тестовых досок.
2. 5-10 независимых оценщиков (члены команды + целевая аудитория) оценивают каждую пару по шкале 1-5: насколько похожи вайбы владельцев.
3. Усреднить человеческие оценки по каждой паре → `human_score ∈ [1, 5]`.
4. Вычислить `model_score = cos(profile_A, profile_B)` для тех же пар.

**Метрика:** Spearman's ρ между `human_score` и `model_score`.

**Target:** ρ ≥ 0.50. Значение выше 0.5 считается сильной корреляцией для субъективных оценок; ρ < 0.3 сигнализирует, что модель не ловит то, что люди воспринимают как «похожий вайб», - в этом случае требуется пересмотр весов фьюжна или включение дополнительных сигналов.

### 2.5. HNSW recall vs brute-force

**Гипотеза:** приближённый поиск в pgvector HNSW не должен терять существенно больше 5% true-nearest соседей относительно точного перебора.

**Методика:**
1. Построить HNSW-индекс на 10 000 синтетических профилей с параметрами из ML-требований (m=16, ef_construction=128).
2. Выбрать 500 случайных query-векторов.
3. Для каждого: получить top-50 через HNSW (ef_search=64) и top-50 через exact cosine distance.
4. Вычислить recall как `|HNSW_top50 ∩ exact_top50| / 50`.

**Метрика:** `ann_recall@50 = mean recall`

**Target:** ≥ 0.95. При падении ниже 0.92 - увеличить `ef_search` до 128 и повторить.

## 3. Offline retrieval метрики

Применяются как на proxy-данных до запуска, так и на реальных данных в рамках regular evaluation pipeline после накопления ≥ 10k пользователей.

| Метрика | Формула | Назначение | Target (pre-launch, proxy) |
|---|---|---|---|
| **Recall@10** | `\|relevant ∩ top10\| / \|relevant\|` | Насколько часто модель находит хотя бы одного релевантного кандидата в топ-10 | ≥ 0.60 |
| **Recall@50** | то же, k=50 | Основная метрика для задачи feed-ленты | ≥ 0.85 |
| **MRR@50** | `mean(1 / rank_of_first_relevant)` | Скорость нахождения первого релевантного | ≥ 0.25 |
| **NDCG@50** | с нормализованной релевантностью | При наличии градуированной релевантности (human_score 1-5) | ≥ 0.45 |
| **Coverage@1000** | `\|распределение top-1 по пользователям\| / N` | Защита от «универсального любимчика» - одного профиля, попадающего в топ всем | ≥ 0.30 |
| **Intra-list diversity** | `1 - mean(cos(p_i, p_j))` в выдаче из 50 | Разнообразие рекомендаций - важно для exploration | ≥ 0.40 |

**Определение «релевантного»** на pre-launch этапе:
- Для 2.2 (aesthetic cluster): релевантны профили из того же стилистического кластера.
- Для 2.4 (human-agreement): релевантны профили с `human_score ≥ 4`.

После запуска релевантность переопределяется через позитивные исходы матчей (свайп-лайк с взаимным ответом → диалог).

## 4. Online-метрики (после запуска MVP)

Измеряют реальную бизнес-ценность. Собираются через event-логи в Kafka/ClickHouse, считаются еженедельно.

### 4.1. Primary метрика

**Match-to-chat rate** - доля взаимных лайков, приводящих к диалогу (≥ 5 сообщений с обеих сторон в течение 72 часов).

`m2c_rate = chats_started / mutual_likes`

**Target MVP:** ≥ 0.25. Референсные значения в индустрии (Tinder, Bumble) - 0.10-0.15; гипотеза Uneemi в том, что визуально обоснованный матчинг даст более высокий m2c за счёт «готовой темы для разговора» (доска как icebreaker).

### 4.2. Secondary метрики

| Метрика | Формула | Target MVP |
|---|---|---|
| **Swipe-right rate** | `right_swipes / total_swipes` | 0.15-0.25 (слишком высокий = отсутствие селекции; слишком низкий = плохие рекомендации) |
| **Mutual like rate** | `mutual_likes / right_swipes` | ≥ 0.08 |
| **Avg chat length (24h)** | сообщения / диалог | ≥ 15 |
| **D7 retention** | `\|DAU_day7 ∩ new_users\| / \|new_users\|` | ≥ 0.25 |
| **D30 retention** | то же для 30 дней | ≥ 0.12 |
| **Feed completion rate** | `seen_profiles / feed_size` | ≥ 0.70 |

### 4.3. Guardrails

Метрики, ухудшение которых блокирует раскатку модели независимо от роста primary:

| Guardrail | Условие блокировки |
|---|---|
| **Report rate** | рост более чем на +20% относительно baseline |
| **Block rate** | рост более чем на +15% |
| **Time-to-first-match** | рост более чем на +30% |
| **p99 latency матч-запроса** | > 500 мс (аварийный порог отката guardrails; целевой SLO p99 <= 200 мс, см. раздел 6) |
| **Error rate ML-сервиса** | > 0.5% |

### 4.4. Lift над baseline'ами

После запуска - постоянное сравнение с двумя baseline'ами:

1. **Random baseline:** случайная выдача из всех профилей того же города/пола. Ожидаемый lift по m2c_rate: ≥ +100%.
2. **Tag-based baseline:** матчинг по пересечению самозаявленных интересов/тегов (упрощённая модель анкетных приложений). Ожидаемый lift: ≥ +40%.

Эти baseline'ы реализуются как альтернативные arms в A/B-тесте (см. раздел 5).

## 5. План A/B тестирования

### 5.1. Основной эксперимент после запуска MVP

**Гипотеза:** ML-матчинг на основе SigLIP 2 эмбеддингов + quiz-фьюжна даёт m2c_rate не меньше +40% относительно tag-based baseline на горизонте 14 дней.

**Дизайн:**
- **Control (A):** tag-based матчинг (пересечение самозаявленных интересов).
- **Treatment (B):** ML-матчинг (SigLIP 2 + quiz, 775d profile vector).
- **Split:** 50/50, рандомизация на уровне user_id, sticky assignment.
- **Unit of analysis:** user (не session).

**Primary:** m2c_rate. **Secondary:** D7 retention, avg chat length. **Guardrails:** report rate, block rate, latency.

**Расчёт выборки:**
- Baseline m2c_rate (оценка): 0.18.
- MDE (minimum detectable effect): +40% относительно baseline → 0.252.
- Alpha = 0.05, Power = 0.80, two-sided.
- Требуемая выборка ≈ 1 600 mutual_likes на arm ≈ 8 000 активных пользователей на arm.
- При DAU 500 и ~10% пользователей с mutual_likes в день - ориентир по длительности: 14-21 день.

**Решающее правило:**
- Primary significant (p < 0.05) + positive → раскатка Treatment на 100%.
- Primary не significant, но guardrails OK → продление теста до 4 недель.
- Любой guardrail нарушен → немедленный откат на Control.

### 5.2. Последующие эксперименты

После стабилизации baseline - серия iterated experiments над гиперпараметрами фьюжна:

| Эксперимент | Что тестируется | Гипотеза |
|---|---|---|
| **Weights fusion** | w_board ∈ {0.6, 0.7, 0.8}, w_quiz ∈ {0.2, 0.3, 0.4} | Оптимальный вес эстетической составляющей |
| **Board composition** | w_meme/w_aesthetic ∈ {0.3/0.7, 0.4/0.6, 0.5/0.5} | Какая модальность даёт больше сигнала |
| **Cold start threshold** | Min images to show in feed ∈ {10, 15, 20} | Trade-off между onboarding-скоростью и качеством |
| **Confidence penalty** | Формула штрафа за малое число изображений | Как избежать matching до готовности доски |
| **HNSW ef_search** | {32, 64, 128} | Trade-off между latency и recall в продакшене |

## 7. Итерация и пересмотр

Метрики пересматриваются после каждой фазы:
- **После sanity-проверки** - фиксируется baseline модели, возвращение к этапу невозможно без переобучения.
- **После proxy-тестов** - принимается решение go/no-go на запуск MVP. При провале ≥ 2 из 5 proxy-тестов - возврат к выбору модели или пересмотру весов фьюжна.
- **После 4 недель live** - накопленные данные позволяют перейти от proxy-релевантности к реальной (исходы матчей), offline-метрики пересчитываются на свежих данных.
- **После 3 месяцев live** - появляется достаточно позитивных пар для исследования fine-tuning SigLIP 2 на задачу Uneemi (contrastive finetune на парах «матч → диалог vs матч → без диалога»). Это отдельный R&D-трек.
