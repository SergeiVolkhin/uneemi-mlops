# Манифест разработки ML-системы Uneemi (критерий 2)

Одностраничный манифест. Заявляемый уровень зрелости: **2** (полный MLOps).

## 1. Предыстория
Заказчик - продукт знакомств Uneemi (RU/CIS). Проблема классических анкетных
приложений: матчи по самозаявленным тегам слабо конвертируются в живой диалог,
пользователи быстро выгорают. Нужна система, подбирающая собеседника по реальному
"вайбу" - визуальной доске и короткой анкете - и непрерывно улучшающаяся на данных.

## 2. Ценностное предложение
Матч, обоснованный визуальным сходством, даёт готовую тему для разговора (доска как
icebreaker) и повышает долю состоявшихся диалогов. Для бизнеса это рост вовлечённости
и удержания; для пользователя - меньше пустых матчей, больше релевантных собеседников.

## 3. Цели
- Предсказывать вероятность диалога для пары профилей и ранжировать ленту по ней.
- Обслуживать полный жизненный цикл модели: от данных до вывода устаревшей модели из
  эксплуатации через переключение трафика.
- Автоматизировать переобучение при дрифте/деградации (continuous training).

## 4. Решение
Обучаемая модель - классификатор матча: вход 775d признаков пары (768 board SigLIP +
7 quiz), выход вероятность матча. SigLIP 2 (ONNX) - слой извлечения признаков
(фичестор). Фичи в Feast (offline parquet + online Redis), эксперименты и реестр в
MLflow, оркестрация в Airflow (3 DAG: фичи, обучение, мониторинг), сервинг в FastAPI
с горячим переключением Production-версии, мониторинг Prometheus+Grafana+Evidently,
CI/CD в GitHub Actions, инфраструктура как код (docker compose, terraform-скелет VPS).
Не входит в MVP: дообучение (fine-tune) SigLIP, мобильный клиент, антифрод,
музыкальная модальность (жанровый словарный счёт без обучаемой модели). Профильный
вектор осознанно сокращён до board (768) + quiz (7) = 775d, чтобы демонстрируемый
MLOps-цикл строился на признаках, которые реально обучаются, а не на счётчике по словарю.
**Заявленный уровень зрелости: 2** (версионирование + CI/CD + фичестор + сервинг +
мониторинг + система экспериментов MLflow + оркестратор Airflow).

## 5. Осуществимость
Ядро готово: ONNX-инференс SigLIP 2 прошёл sanity (ImageNet zs 78.48%, XM3600 RU
73.82%) и performance-бенчмарки. MLOps-слой собирается из открытых компонентов,
поднимается одной командой на Windows+Docker, переносим в облако (12-factor). Ресурсы:
один разработчик, dev-машина 12C/32GB, целевой VPS 4 vCPU / 8GB.

## 6. Данные
- Источник признаков: доски пользователей (картинки) -> SigLIP board-эмбеддинг; анкета
  -> 7 quiz-признаков. Offline-набор - в фичесторе (Feast), online (прод) - в Redis.
- Процесс разметки (реальный, целевой): позитивный исход = взаимный лайк, перешедший
  в диалог (>= 5 сообщений с обеих сторон за 72ч) -> положительная пара; взаимный лайк
  без диалога -> отрицательная. Разметка копится из event-логов после запуска.
- Текущий статус: реальных исходов нет, поэтому для демонстрации используются
  СИНТЕТИЧЕСКИЕ помеченные пары (синтетика для демонстрации) с честным процессом:
  P(match) растёт с косинусной близостью реальных SigLIP-профилей плюс шум.

## 7. Метрики
Приоритет бизнес-показателей (отражают цели п.3): главная - m2c_rate (match-to-chat,
target >= 0.25); вторичные - swipe-right rate, mutual like rate, D7/D30 retention.
Полный перечень и пороги - `docs/metrics_and_benchmarks.md`, перевод в ML-метрики -
`docs/goal_and_metrics.md`.

## 8. Оценка качества модели
- Offline: ROC-AUC (главный гейт), Precision/Recall/F1 на holdout; для retrieval -
  Recall@K. Считаются в training_pipeline, логируются в MLflow.
- Online: m2c_rate и guardrails (report/block rate, p99 latency, error rate),
  live ROC-AUC на свежей разметке в monitoring_pipeline.

## 9. Подбор модели
Итеративно: базовая модель champion (логистическая регрессия) и претендент challenger
(MLP). На каждом прогоне обе обучаются и сравниваются на holdout; претендент попадает
в Production только пройдя гейт (AUC >= порога И выше текущего champion). Дальнейшие
итерации - гиперпараметры фьюжна (см. `docs/metrics_and_benchmarks.md`, раздел 5.2).

## 10. Инференс
Реального времени (online): FastAPI отдаёт вероятность матча по паре профилей,
признаки берёт из online-store Feast (Redis, low-latency). Извлечение board-
эмбеддингов (SigLIP) - батчами в feature_pipeline (тяжёлая операция, не на горячем пути).

## 11. Обратная связь
Источники: исходы матчей (лайк -> диалог), поведенческие метрики ленты, дрифт фич и
падение live-качества. Мониторинг при срыве порогов триггерит переобучение
(continuous training). **Используем MDD: Да** (см. `docs/mdd/ADR-0001-latency.md`).
